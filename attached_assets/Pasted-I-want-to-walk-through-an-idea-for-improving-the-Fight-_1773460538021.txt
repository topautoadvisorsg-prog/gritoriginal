I want to walk through an idea for improving the Fight Analyst AI so it scales well when we have a lot of users. This is just the direction we’re thinking about — I want your input on the cleanest way to implement it with the current architecture.

There are a few parts to it.

1. Suggested questions (UX improvement)

Most users will ask the same types of questions about a fight. Things like:

• Who has the striking advantage?
• Who has the grappling advantage?
• What is the likely path to victory?
• How do their last 5 fights compare?
• Does reach or size play a big role in this matchup?
• Who has the cardio advantage?

So under the Fight Analyst AI chat we could show a small set of clickable “Suggested Questions”. When a user clicks one it just sends that prompt to the AI chat. Users should still be able to ask completely custom questions as well — this would just make it faster for people to interact with the AI.

2. AI answer caching (so we don’t regenerate the same answers repeatedly)

Since a lot of users will ask the same questions, we want to avoid hitting the AI model every time if the same question was already answered for that fight.

The idea is to implement a per-fight cache.

Conceptually something like:

fightId
question
questionEmbedding (or similarity key)
answer
createdAt

Flow would be:

• User asks a question
• System checks existing stored questions for that fight
• If a similar question already exists, return the stored answer immediately
• If not, generate the AI response and store it for future reuse

This would reduce a huge number of repeated AI calls for popular fights.

3. Similarity matching instead of exact matching

We shouldn’t rely on exact text matches because people phrase things differently.

Example:

“Who has the striking advantage?”
“Which fighter is the better striker?”
“Who wins the stand-up battle?”

These are basically the same question.

So the system should check similarity (embeddings or some kind of semantic comparison) to determine whether the question already exists in the cache.

4. Cache should belong to the fight, not the fighter

Since these questions are about matchups, the cache should be scoped to the fight itself.

So each fight effectively builds its own small “AI knowledge cache” over time.

Fight
├ Fighter A
├ Fighter B
└ AI cache (questions + answers)

5. Storage and long-term data

Because these cached answers are just text, storage should be very small. Because of that we probably don’t need to delete them after the fight.

Instead we could keep them as historical AI analysis tied to the fight. That could later let us compare things like pre-fight AI analysis vs actual fight outcomes.

If there’s a simple way to mark them as archived after the fight instead of deleting them, that might be useful.

6. Practical limits

We probably only need something like 50–100 cached questions per fight max. Realistically most fights will only generate maybe 15–30 meaningful analysis questions.

So the goal is:

• Suggested questions for quick interaction
• Cached answers for repeated questions
• Custom questions still allowed
• Cache scoped to each fight

Before we implement anything, I’d like your opinion on this architecture.

Does this approach fit cleanly with the current AI chat system and backend we already built? If you were designing it, would you handle the caching or similarity matching differently?

Curious what you think is the cleanest and most scalable way to implement it.
