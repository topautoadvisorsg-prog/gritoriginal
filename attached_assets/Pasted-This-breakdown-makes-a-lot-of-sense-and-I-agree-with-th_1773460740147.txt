This breakdown makes a lot of sense, and I agree with the rollout approach you proposed.

Your point about embeddings is valid — for the stage we’re at, it’s better to avoid adding unnecessary API calls and complexity. Starting with exact-match caching for suggested questions and using those to seed the cache sounds like the right 80/20 solution.

So if I’m understanding your recommendation correctly, the phased approach would be:

Phase 1 (initial implementation)

• Add the ai_fight_qa_cache table
• Seed each fight with a fixed list of suggested questions (stored in the DB)
• When a suggested question is clicked, check the cache by questionKey
• If cached → return the stored answer (stream it so the UI stays consistent)
• If not cached → generate via AI and store it
• Track hitCount so we can see which questions are actually being used

Frontend side:

• Render the suggested questions as clickable chips in Fight Analyst AI when in fight-context mode
• Clicking a chip sends it as a normal message through the existing AI handler

Lifecycle:

• When a fight becomes Completed, mark those cache entries as isArchived = true
• Archived entries remain for historical analysis but are not reused for future responses

Phase 2 (later, based on usage data)

• Add embedding similarity for custom questions
• Generate embeddings lazily for new questions when needed
• Allow similar custom questions to hit the same cached answer

That approach sounds clean and keeps the implementation simple while still leaving room for the smarter similarity layer later.

2. For the hitCount tracking, do you think we should also capture basic analytics like how many times the AI chat is opened per fight, so we can measure engagement with the feature?
