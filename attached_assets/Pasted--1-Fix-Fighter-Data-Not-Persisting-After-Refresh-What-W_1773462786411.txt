#1 - Fix Fighter Data Not Persisting After Refresh
What & Why
Fighter data imported via the admin UI disappears after every page refresh because the write routes are registered on the wrong server. The app runs two Express servers:

User-server (port 3001) — handles /api/* routes, where Vite proxies all frontend API calls
Admin-server (port 3002) — handles /api/admin/* routes only
All fighter write routes (POST /api/fighters/bulk, PUT /api/fighters/:id, DELETE /api/fighters/:id, POST /api/fighters, POST /api/fights/bulk, PUT /api/fights/:id, DELETE /api/fights/:id, POST /api/fighters/:id/relink, POST /api/fighters/:id/import-history) are registered exclusively on the admin-server (port 3002). The Vite proxy only sends /api/admin/* requests to port 3002 — everything else goes to port 3001 (user-server), which has no write routes for fighters.

When the import flow calls POST /api/fighters/bulk, the request hits port 3001 and gets a 404. FighterDataContext.addFighters() catches the failure and updates local React state only. Fighters appear in the UI but are never written to PostgreSQL. After refresh, the React state resets and the empty DB produces zero fighters.

Done looks like
Importing fighters via the admin import page saves them to the database
After browser refresh or server restart, imported fighters are still present
Fighter edit/save and delete operations also persist correctly
The DB fighters table has a non-zero row count after import
No changes to auth guards — all write routes remain admin-only
Out of scope
Fixing the same routing gap for other entities (events, news, etc.) — scoped to fighters and fight history only
Migrating the two-server architecture to a single server
Any schema or frontend changes
Tasks
Register admin fighter write routes on the user-server — Import registerAdminFighterRoutes in server/user-server.ts and call it alongside the existing registerFighterRoutes call. This exposes all fighter write operations (POST /api/fighters, POST /api/fighters/bulk, PUT /api/fighters/:id, DELETE /api/fighters/:id, POST /api/fighters/:id/relink, POST /api/fighters/:id/import-history, PUT /api/fights/:id, DELETE /api/fights/:id, POST /api/fights/bulk) on port 3001, which is where the Vite proxy sends them. All requireAdmin guards remain intact. Restart the workflow and verify by importing a fighter and confirming it persists after refresh and that SELECT COUNT(*) FROM fighters returns > 0.
