Objective

Perform end-to-end behavioral testing of the platform using fantasy/test data only.
Do not spend time crafting real or meaningful content.

The goal is to validate system logic, routing integrity, payload handling, locking rules, scoring, and module interaction.

Step 1 — Generate Test Fighters

Create 24–30 fighter profiles.

Each fighter must have:

Name (can be fake/test names)

Stats fields

Record fields

Any schema payload fields required by the database or API

Important rules:

If a schema field exists → fill it.

If you don’t know what value to put → use:

"test"

"test1"

"123"

Or any placeholder value.

Do NOT worry about realism.

The purpose is payload completeness.

Step 2 — Create Full Fight Card

Using the fighters created above:

Construct approximately 12 fights.

Group them into a complete fantasy event card.

You may invent:

Matchups

Methods

Rounds

Odds (if required)

Metadata fields

Nothing must be real.

This is disposable testing data.

Step 3 — Create Test User

Create a test user account.

Test user should be able to:

View event card

Navigate event → fight detail → pick flow

Submit picks

Observe lock behavior

Step 4 — Core Flow Simulation

Simulate a real user interaction sequence:

Login

Navigate to Event Page

Open Fight Card

Enter Fight Detail Page

Make picks

Test analysis AI section (premium gating must behave correctly)

Step 5 — Event Close Behavior Testing

When fights are finalized:

Verify:

✅ Picks are locked
✅ Scoring is calculated inside transaction logic
✅ Prestige key system triggers only once per clean sweep
✅ Leaderboard updates correctly
✅ Badge audit logs are written
✅ Duplicate processing is impossible

Simulate:

Multiple finalize requests if possible.

Step 6 — Payload and Schema Stress Test

For fighters, events, and picks:

Fill every payload field even if value is dummy.

Do not leave optional fields empty if schema expects data.

Goal is to verify backend validation and serialization.

Step 7 — Data Cleanup Policy

All generated test data will be deleted after validation.

No test data should be considered production information.

Step 8 — Quality Philosophy

Focus on system correctness, not content quality.

Use fantasy data freely.

If something can be made better during testing, improve it.

Always attempt to break edge cases.

Deliverable Checklist

Confirm:

Routing works

Auth role hydration works

Sidebar navigation is correct

Admin modules load

Fight lifecycle works

Locking logic is enforced

Socket/chat streams do not crash

AI gating behaves correctly

If possible, also perform a “user chaos simulation”:

Rapid clicking navigation

Multiple pick submissions

Simultaneous finalize attempts

Long chat messages

Logout/login mid-stream
